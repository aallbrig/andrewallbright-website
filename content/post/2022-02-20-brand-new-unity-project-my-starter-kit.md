Testing is a core tenant I believe helps me produce high quality software. In order to have faith in a test suite that is assembled is to constantly verify every code change that is introduced to the system.

Github Actions (<a rel="noreferrer noopener" href="https://docs.github.com/en/actions" target="_blank">docs</a>) is a wonderful workflow based system that can easily integrate with a Github project. Since Github is my version control repository of choice, it is natural for me to adopt it.

I lean heavily on Game.ci (<a rel="noreferrer noopener" href="https://game.ci/" target="_blank">docs</a>), which is an open source project that uses containers and wizardry to produce Unity game builds.

Here is my default github actions definition for a unity project. There are assumptions that must be true

<ul class="wp-block-list">
  <li>
    The unity project files are a subdirectory (aka subfolder) of a directory named <code>unity</code>. This directory will be the watch directory. This isolation allows this workflow to only build if unity project files are touched (aka dirty).
  </li>
  <li>
    This example uses my latest unity project <code>global-game-jam-2022</code>. If you are using vim, you can change the project name with this snippet: <code>%s/UNITY_PROJECT_DIRECTORY_NAME/   your desired name here (remove spaces)   /g</code>
  </li>
  <li>
    Github Pages (<a rel="noreferrer noopener" href="https://docs.github.com/en/pages" target="_blank">docs</a>) is enabled and is targeting a <code>docs</code> subdirectory in the root of the project folder. That is ultimately where some unity webGL build files will go.
  </li>
  <li>
    This file needs to go in the <code>.github/workflows</code> directory. If you do not have it, run <code>mkdir -p .github/workflows</code> to create it.
  </li>
</ul><figure class="wp-block-image size-large">
